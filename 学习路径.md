## 第二阶段：Redis（第 2 周前半，3-4 天）

> 目标：理解 Redis 在本项目中的三个用途：缓存库存、Lua 脚本扣减、分布式锁

### Day 1：Redis 基础

**必学内容：**

1. **5 种数据类型**
   - String（本项目用这个存库存：`slot:stock:1` → `20`）
   - Hash、List、Set、ZSet（了解即可）

2. **基本操作**
   - `SET key value` / `GET key` / `DECR key`
   - `EXPIRE key seconds`（设置过期时间）
   - 操作：`SET slot:stock:1 20 EX 172800`（设置库存 20，2 天后过期）

3. **为什么 Redis 快？**
   - 纯内存操作
   - 单线程（避免锁竞争）
   - IO 多路复用（epoll/kqueue）

### Day 2：Redis + Spring Boot

**必学内容：**

1. **RedisTemplate 配置**（对照 `RedisConfig.java`）
   - `StringRedisSerializer`：key 用字符串序列化
   - `GenericJackson2JsonRedisSerializer`：value 用 JSON 序列化

2. **RedisTemplate 操作**
   ```java
   // 读写（来自 SlotService.java）
   redisTemplate.opsForValue().set(key, value, 2, TimeUnit.DAYS);
   redisTemplate.opsForValue().increment(key, 1);
   
   // 执行 Lua 脚本（来自 ReservationService.java:63）
   redisTemplate.execute(deductStockScript, Collections.singletonList(key));
   ```

3. **缓存预热**（对照 `SlotService.java`）
   - `@Scheduled(cron = "0 0 0 * * ?")`：每天凌晨执行
   - 逻辑：从 MySQL 查库存 → 写入 Redis，设置 2 天过期
   - 兜底：请求时发现 Redis 没有 key，自动触发预热

### Day 3：Redis Lua 脚本

**必学内容：**

1. **为什么需要 Lua 脚本？**
   - Java 里先 GET 再 DECR = 两步操作，中间可能被其他线程插入
   - Lua 脚本在 Redis 服务端原子执行，不会被打断

2. **阅读项目 Lua 脚本**（`src/main/resources/lua/deduct_stock.lua`）
   ```lua
   local stockKey = KEYS[1]
   local currentStock = redis.call('GET', stockKey)
   
   if currentStock == false then
       return -1              -- Key 不存在，需要预热
   end
   
   currentStock = tonumber(currentStock)
   if currentStock <= 0 then
       return 0               -- 库存不足
   end
   
   redis.call('DECR', stockKey)
   return 1                   -- 扣减成功
   ```

3. **Spring 加载 Lua 脚本**（对照 `RedisConfig.java:27`）
   ```java
   DefaultRedisScript<Long> script = new DefaultRedisScript<>();
   script.setLocation(new ClassPathResource("lua/deduct_stock.lua"));
   script.setResultType(Long.class);
   ```

### Day 4：缓存穿透 / 击穿 / 雪崩（面试高频）

**必学内容：**

1. **缓存穿透**：查一个数据库里不存在的 key，每次都打到 MySQL
   - 解决：布隆过滤器 / 缓存空值

2. **缓存击穿**：热点 key 过期的瞬间，大量请求打到 MySQL
   - 解决：互斥锁 / 永不过期 + 后台刷新

3. **缓存雪崩**：大量 key 同时过期，MySQL 压力骤增
   - 解决：过期时间加随机值 / 多级缓存

## 第三阶段：Redisson 分布式锁（第 2 周后半，2 天）

> 目标：理解分布式锁的原理、Redisson 的实现、本项目的锁策略

### Day 1：分布式锁原理

**必学内容：**

1. **为什么需要分布式锁？**
   - 单机环境用 `synchronized` 就够了
   - 分布式环境下，多个 JVM 实例，synchronized 失效

2. **Redis 分布式锁原理**
   - `SET key value NX EX 5`：key 不存在才设置，5 秒过期
   - 问题：锁误删（A 的锁过期了，B 拿到锁，A 执行完删了 B 的锁）
   - 解决：value 存 UUID，删除时验证是否是自己的锁

3. **Redisson 的封装**
   - 底层用 Redis Hash 结构实现
   - 支持可重入（同一线程可多次加锁）

### Day 2：Redisson 在本项目中的应用

**必学内容：**

1. **tryLock 三参数**（对照 `ReservationService.java:51`）
   ```java
   lock.tryLock(waitTime, leaseTime, TimeUnit.SECONDS)
   // waitTime=3：最多等 3 秒获取锁
   // leaseTime=5：锁持有 5 秒后自动释放（防止死锁）
   ```

2. **看门狗（Watch Dog）机制**
   - 如果业务执行超过 5 秒，Redisson 会自动续期
   - 默认续期到 30 秒（每 10 秒续一次）
   - 只有指定了 leaseTime 才会关闭看门狗（本项目指定了 leaseTime=5，所以不会续期）

3. **锁粒度设计**（本项目的关键）
   ```
   锁 key：lock:reservation:{userId}:{slotId}
   粒度：用户级别
   效果：
     - 同一用户对同一场次：串行执行（防重复提交）
     - 不同用户之间：互不影响（高并发）
     - 同一用户对不同场次：互不影响
   ```

4. **finally 释放锁**（对照 `ReservationService.java:112`）
   ```java
   if (lock.isHeldByCurrentThread()) {
       lock.unlock();
   }
   ```

## 第四阶段：RabbitMQ（第 3 周前半，3 天）

> 目标：理解消息队列的作用、Exchange/Queue/Binding 概念、本项目的异步处理流程

### Day 1：RabbitMQ 基础概念

**必学内容：**

1. **为什么要用消息队列？**
   - **解耦**：订单系统和库存系统不直接依赖
   - **削峰**：10000 个请求进来，队列慢慢消费
   - **异步**：用户下单后立即返回，后台慢慢处理

2. **核心组件**
   ```
   Producer（生产者）→ Exchange（交换机）→ Queue（队列）→ Consumer（消费者）
   ```
   - **Exchange 类型**：Direct / Topic / Fanout / Headers
   - **Routing Key**：路由规则，决定消息发到哪个队列
   - **Binding**：Exchange 和 Queue 的绑定关系

3. **本项目的配置**（对照 `RabbitMQConfig.java`）
   ```java
   // Topic 交换机
   new TopicExchange("gym.reservation.exchange", true, false);
   // 持久化队列
   QueueBuilder.durable("gym.reservation.queue").build();
   // 绑定：exchange → routingKey → queue
   BindingBuilder.bind(queue).to(exchange).with(routingKey);
   ```

### Day 2：Spring Boot + RabbitMQ

**必学内容：**

1. **发送消息**（对照 `ReservationService.java:95`）
   ```java
   rabbitTemplate.convertAndSend(EXCHANGE, ROUTING_KEY, reservation);
   ```

2. **接收消息**（对照 `ReservationConsumer.java:21`）
   ```java
   @RabbitListener(queues = "${gym.reservation.queue.queue}", concurrency = "5")
   public void handleReservation(Message message, Channel channel) {
       // 1. 解析消息
       // 2. 处理业务
       // 3. ACK 确认
       channel.basicAck(deliveryTag, false);
   }
   ```

3. **消息确认机制**
   - `ACK`：处理成功，从队列移除
   - `NACK`：处理失败，重新入队
   - `basicNack(deliveryTag, false, true)`：第三个参数 true 表示重新入队

4. **降级策略**（对照 `ReservationService.java:98`）
   ```java
   try {
       rabbitTemplate.convertAndSend(EXCHANGE, ROUTING_KEY, reservation);
   } catch (Exception e) {
       // MQ 不可用，降级为同步处理
       processReservation(reservation);
   }
   ```

### Day 3：消息队列可靠性

**必学内容：**

1. **消息不丢失的三要素**
   - **生产者**：开启确认模式（`publisher-confirm-type: correlated`）
   - **Broker**：队列和消息持久化（`durable: true`）
   - **消费者**：手动 ACK（`acknowledge-mode: manual`）

2. **幂等性**（防重复消费）
   - 本项目在 `processReservation()` 中做了二次幂等检查
   - 数据库唯一索引 `UNIQUE KEY uk_slot_user (slot_id, user_id)` 兜底
